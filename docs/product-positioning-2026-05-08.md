# driftpet Product Repositioning

## Why Reposition

The weak version of driftpet is "a pet that reminds you to focus."

That framing does not justify itself. If the user already knows they are distracted, another reminder is redundant. If the user can sort the mess out entirely in their head, a generic summary is also redundant.

The stronger framing is:

> driftpet is a cognitive desk companion that catches drift and holds the loose thread until you can return to it.

Its job is not to think for you. Its job is to keep your thread from dissolving when your attention forks.

## Product Promise

driftpet should feel like a small living presence on the desktop that:

1. notices when a thread is slipping,
2. gives you a low-friction place to dump the mess,
3. compresses it into one next move,
4. remembers just enough context to help you re-enter later,
5. stays quiet when it has nothing useful to add.

## What It Is Not

- Not a generic productivity reminder
- Not a habit tracker
- Not a chatbot that always has something to say
- Not a passive surveillance tool that watches everything you do
- Not a second brain that tries to replace your own

## The Three Pillars

### 1. Presence

The pet should feel alive because it has stance, not because it is noisy.

Desired qualities:

- quiet by default
- lightly attentive
- visibly in a state, not just looping animation
- emotionally non-judgmental

The user should feel "it is here with me" rather than "it is interrupting me again."

### 2. Catch

The value appears at the exact moment the user's thread starts to fray.

The capture path must stay:

- fast
- forgiving
- tolerant of messy input
- better than forcing the user to reorganize thoughts alone before acting

The pet earns its place when dumping chaos into it feels easier than mentally holding the whole knot together.

### 3. Return

The pet should help the user return to the main thread later without replaying useless noise.

That means:

- resurfacing only memories with real value,
- allowing empty recall when nothing good matches,
- maintaining continuity around the user's recent thread,
- preferring one clean re-entry point over many weak associations.

## "Alive" In Practice

For driftpet, aliveness should come from these behaviors:

- It has a current stance toward the user's thread: guarding, waiting, dozing, leaning in.
- It reacts differently when the user is settled, drifting, or explicitly asking for help.
- It speaks like a companion holding a place in line, not a dashboard warning light.
- It remembers the thread it last helped preserve, so the relationship feels continuous.
- It remains restrained; silence is part of the product.

## Existing Surfaces That Already Support This

The current app already has the right anchors:

- `focus / sleep` mode
- hourly surface budget
- manual chaos-reset workbench
- related-memory recall
- active-card presence in the renderer shell
- idle / remark copy surfaces in the pet bubble

This means the next step is not a new feature explosion. It is to tune behavior and voice so the pet feels like a living thread-keeper rather than a floating summarizer.

## Near-Term Product Direction

Near-term work should bias toward:

- calmer, more specific pet voice
- stronger distinction between "quietly present" and "actively helping"
- better thread continuity in related recall
- fewer but better interruptions
- visible state transitions that reflect attention posture

## Non-Goals For This Reposition

- Do not turn driftpet into a general companion chat app.
- Do not add passive sensing just to fake intelligence.
- Do not make the pet louder to simulate aliveness.
- Do not widen scope beyond drift capture, compression, and re-entry.

## Test For Future Decisions

When considering a new feature, ask:

> Does this help the pet catch, hold, or restore a user's thread better than their own head can do unaided in that moment?

If the answer is no, it is probably outside driftpet's real product identity.
