# 线程回报模板

适用：业务线程 / 测试线程 / QA 验收线程。整段必须用**可直接复制的 Markdown 代码块**输出。

适用：L1 / L2 / L3 接入级别。

---

## 标准回报模板

```md
执行线程：
- <线程名称>

任务ID：
- <MMDDTxxx>

状态：
- 执行中 / 待验收 / 已通过 / 未通过 / 阻塞 / 作废

是否进行QA验收：
- 是 / 否

QA说明：
- 无 / 当前任务结果暂不进入QA验收，待总控确认后再决定是否派发QA验收。

files：
- <本轮改动文件，或"无代码改动">

action：
- <本轮实际做了什么>

verify：
- <运行了什么命令；贴关键输出 + exit code>

done：
- <当前真正完成了什么（对照派发的 acceptance 逐条说明）>

blockers：
- <还有什么阻塞；没有则写"无">

最小下一步：
- <若状态为阻塞，写最小恢复动作；否则写"无">

commit：
- <commit id> / 无

提交信息：
- <commit message> / 无
```

---

## 必填字段

- `执行线程` `任务ID` `状态` `是否进行QA验收`
- `verify`（Ω6 可验证）：贴命令 + exit code 或具体输出，不允许"已自测通过"等无证据描述
- `done`：必须对照派发文本的 `acceptance` 逐条回应

---

## 7 个固定状态

只允许使用这 7 个：

1. `待执行`
2. `执行中`
3. `待验收`
4. `已通过`
5. `未通过`
6. `阻塞`
7. `作废`

规则：
- 业务线程通常结束在 `待验收`
- L2 下 QA 最终输出 `已通过 / 未通过 / 阻塞`
- L1 下总控直接判断，无 QA 中转
- 被替代、取消、无需继续 → 统一记 `作废`

## QA 固定话术（L2+）

### 当前还不进入 QA

```md
是否进行QA验收：
- 否

QA说明：
- 当前任务结果暂不进入QA验收，待总控确认后再决定是否派发QA验收。
```

### 当前已经进入 QA

```md
是否进行QA验收：
- 是

QA说明：
- 无
```

## commit 写法

### 需要提交代码时

```md
commit：
- <commit id>

提交信息：
- <commit message>
```

### 不需要提交代码时

```md
commit：
- 无

提交信息：
- 无
```

---

## 阻塞场景的最小下一步

状态为 `阻塞` 时，`最小下一步` 必填，且必须是**单步、可执行**的恢复动作。

反例（禁止）：
- "等待用户确认"（不可执行）
- "重新设计架构"（非最小步）

正例：
- "请用户提供 API key 后重试"
- "把 tests/foo.test.ts:42 的 mock 切到真实 fixture"
- "回退 commit abc123 后重跑 verify"
