# 接入指南

`workflow-portable` 是渐进式接入：**从 L0 开始，只在真的需要更多结构时才升级**。
不要一上来就 L3——多数项目最终稳定在 L0 或 L1。

---

## 决策树（30 秒判断接入哪一级）

```
项目会持续多轮迭代吗？
├─ 否 → 不接入（一次性脚本不需要这套）
└─ 是 ↓

会有 ≥2 个 agent 协作吗？
├─ 否 → L0
└─ 是 ↓

需要独立的验收角色吗？
├─ 否 → L1
└─ 是 ↓

需要机器读派发文本（自动化看板/编排）吗？
├─ 否 → L2
└─ 是 → L3
```

---

## L0 — 单 agent / 个人项目

### 你需要的

- 复制一份 `CLAUDE.md.template` 到项目根，改名 `CLAUDE.md`
- 填好顶部"项目槽位"6 个字段
- 完

### 你不需要的

- 派发模板（自己一个 agent 派给自己很奇怪）
- 回报模板（同上）
- QA 模板
- JSON 任务包

### 第一条任务怎么跑

按 Φ/Ω/Σ/Ξ 自约束。每次回复用六态格式中的一种。Lite 触发条件满足时可跳过完整六态。

### 何时升级 L1

出现这两种情况之一：
- 你开始让多个 agent 协作（比如总控派给 executor）
- 你开始混淆"哪个版本是最新的"

---

## L1 — 多线程，无独立 QA

### 你需要的

L0 的全部 + 启用：
- `templates/task-dispatch.md`
- `templates/thread-report.md`

### 第一天必须先定下来的 6 件事

1. 谁是总控
2. 有哪些业务线程
3. 是否有独立测试线程（通常 L1 阶段：无）
4. QA 由谁判断（L1：总控自己判断）
5. 任务 ID 规则是否沿用 `MMDDTxxx`
6. 项目固定文档放在哪（`task_plan.md` / `progress.md` / `findings.md`）

### 推荐固定文档

- `AGENTS.md` 或 `CLAUDE.md`：项目级规则（来自 template）
- `task_plan.md`：线程边界、任务池、验收口径
- `progress.md`：当前状态、优先级、阻塞
- `findings.md`：风险、冲突、踩坑记录

### 第一条任务的最小闭环

1. 总控按 `task-dispatch.md` 派发
2. 业务线程按 `thread-report.md` 回报
3. 总控读回报，判断状态，决定下一步

如果这个闭环跑不通 → 不要扩规模，先收规则。

### 何时升级 L2

出现这两种情况之一：
- 总控的判断负担过重，需要拆分
- 出现"业务线程自己说自己通过了"的争议

---

## L2 — 多线程 + 独立 QA

### 你需要的

L1 的全部 + 启用：
- `templates/qa-acceptance.md`
- 固定 QA 文档：推荐 `docs/qa-acceptance-report.md`

### 默认主链路

`业务线程 → QA验收线程 → 总控`

测试线程不在默认链路，只在总控明确派发专项测试时介入。

### 关键约束

- QA 文档是**最终验收事实源**，聊天里的口头结论不能覆盖
- 业务线程通常结束在 `待验收`，由 QA 给出最终 `已通过 / 未通过 / 阻塞`
- "免内容验收"模式下，QA 不判断内容，仅确认流程节点

### 何时升级 L3

只有当你需要**程序读派发文本**时才升级（比如要做任务看板、自动化编排、cross-agent handoff）。
否则停在 L2 是稳定区。

---

## L3 — 机器可读编排

### 你需要的

L2 的全部 + 启用：
- `templates/task-packet.json`
- 运行时目录：推荐 `.workflow/tasks/<task-id>.json`

### 派发流程

1. 总控写 `task-dispatch.md` 派发文本
2. 同步生成 `task-packet.json` 副本
3. 编排程序读 JSON，分发给 executor
4. executor 回报按 `thread-report.md`
5. （可选）回报也结构化为 JSON 进运行时

### 注意

- JSON 是 Markdown 的镜像，**不是替代**——人读的派发文本仍然是事实源
- 如果 JSON 和 Markdown 不一致 → 以 Markdown 为准，修 JSON
- 不要跳过 Markdown 派发直接写 JSON——会丢失人读语境

---

## 常见反模式（任何级别都要避免）

### 1. 任务一次派太多
默认一次 1 个。多个任务并行 → 必须显式 `执行顺序：可并行` 且确认互不依赖。

### 2. 测试线程代替 QA
测试线程**不能**给最终结论。这是常见的边界滑动。

### 3. 规则更新只在聊天里说
规则文档先更新，下一轮任务派发必须带 `规则更新提醒`。

### 4. 任务 ID 复用
不同轮次用不同 ID。复用 = 状态混乱。

### 5. 用"我已自测通过"代替证据
回报的 `verify` 字段必须有命令 + exit code 或具体输出。无证据 = 没完成。

### 6. 一上来就 L3
JSON 任务包的复杂度只在多 agent + 自动化编排场景才回本。
L0 → L3 直跳 = 形式主义。

---

## 接入完成的判断标准

L1+ 接入完成 ⇔ 同时满足：

- [ ] 项目规则文档（CLAUDE.md / AGENTS.md）已落地
- [ ] 第一条任务已按主链路跑完
- [ ] 总控能根据回报（L1）/ QA 文档（L2+）继续派下一条
- [ ] 所有线程开始使用统一模板
- [ ] 回报字段完整，没有"已自测通过"无证据描述

任一不满足 → 接入未完成，不要进入 BAU 节奏。
